The on-board firmware runs on a Seeed Studio XIAO ESP32-S3 and is architected around FreeRTOS's dual-core scheduling rather than a conventional single-threaded Arduino loop, which is a deliberate and appropriate choice for a flight computer where sensor timing integrity cannot be allowed to degrade due to I/O latency. One core is dedicated exclusively to sampling the BME280 barometric sensor and BMI160 inertial measurement unit at a fixed 50Hz cadence, packaging each reading into a fixed-size struct and pushing it into a thread-safe FreeRTOS queue; the second core independently drains that queue, serializes each packet into a compact CSV string, transmits it over the 433MHz LoRa Ra-02 radio, and simultaneously appends it to a persistent log file on the on-board SD card. This producer-consumer separation ensures that a slow SD write or radio transmission can never stall or introduce jitter into the sensor sampling rate, which is the correct priority ordering for a system whose primary purpose is producing a clean, evenly-spaced time series. The firmware also treats data integrity as a first-class concern at the hardware level: the IMU's accelerometer and gyroscope full-scale ranges are explicitly configured rather than left to library defaults, ensuring the raw sensor counts are converted to physical units using scale factors that are actually true for the configured hardware state, and the LoRa radio's built-in CRC is enabled so that any packet corrupted in transit by RF interference is discarded by the hardware itself rather than silently propagating a false reading into the ground station. The result is a self-contained flight computer that produces two independent, synchronized copies of the same telemetry — one transmitted live for real-time ground tracking, one written to non-volatile storage as a durable record that survives even if the radio link is lost entirely.
